IoT Design Methodology - Bahga & Madisetti, 10 Steps

JamShield: Jamming-Resilient IoT Sensor Node

Problem. A normal WiFi IoT node has a single radio, so a WiFi jamming attack (deauth flood) silences it completely. Solution. JamShield detects the jam in real time and automatically hops across three radios (WiFi → BLE → ESP-NOW), keeping telemetry alive. A Raspberry Pi hub unifies all channels and serves a live app + MQTT cloud.

Specification

Purpose & Requirements

  • Purpose: keep an IoT sensor reporting during a WiFi denial-of-service (jamming) attack by switching radios automatically.
  • Behaviour: sense ambient light (LDR on ESP32), publish telemetry, detect jamming, fail over to a backup radio, recover to WiFi when the attack stops.
  • Functional requirements: real-time detection (< 500 ms), automatic WiFi → BLE → ESP-NOW failover, no/minimal packet loss (sequence-tracked), MQTT integration, web/mobile app, IoT cloud.
  • Non-functional: low latency, high availability, edge-first (works offline), low power, portable firmware, secure and scoped.
2
Specification

Process

  • Node reads the LDR, builds a packet with a monotonic sequence number, and sends it over the currently active radio.
  • A high-priority detector samples RSSI + MQTT ack-loss every 100 ms and runs a jam state machine.
  • On a confirmed jam the bearer manager fails over to the next radio; on recovery it hops back to WiFi.
  • The Pi receives on any channel, logs it, and republishes a unified feed to the app and cloud.

Domain Model

  • Physical entities: ESP32 node, LDR sensor, ESP32 jammer, Raspberry Pi, user device.
  • Virtual entities / resources: jam-detector FSM, bearer manager, MQTT broker, unified feed, database, PWA, cloud broker.
  • Relationships: the LDR feeds the node; the detector drives the bearer manager; the jammer attacks WiFi; all radios converge on the Pi; the user controls the node through the app.

Specification

IoT Level

  • Level 5-style deployment: constrained nodes at the edge, a coordinator (Raspberry Pi) doing local processing + storage, and cloud for remote monitoring.
  • Edge-first: the whole system works on the local network with no internet; the cloud is an optional mirror.
  • Local processing (detection + failover) lives on the node for low latency; aggregation + app + history live on the Pi; global view lives in the cloud.

Operational View

  • Node firmware: Zephyr RTOS on ESP32 (multi-threaded, priority scheduler).
  • Pi services (auto-start): Mosquitto (MQTT 1883 + WebSocket 9001), receiver, PWA host (:8081), cloud bridge.
  • Protocols: MQTT QoS 1, BLE GATT notify, ESP-NOW 802.11 vendor frames, WebSockets to the app, TLS to cloud.

9
Build

Device & Component Integration

  • Node ESP32: LDR voltage divider on GPIO34 (ADC); WiFi, BLE and ESP-NOW radios enabled; flashed with the Zephyr firmware.
  • Jammer ESP32: associates to the target AP and injects deauth frames (authorised, own-network only).
  • Raspberry Pi: Mosquitto broker, Bluetooth enabled (pi-bluetooth), Python receiver + SQLite, PWA host, and TLS bridge to HiveMQ, all as systemd services.
  • Programming: firmware built with west, flashed over USB with esptool; one launcher (run.ps1) auto-detects ports + Pi IP, patches config, builds and flashes.

Build

Application Development

  • PWA control room (installable web/mobile app): live status (LINK SECURE / ATTACK DEFEATED / DATA LOST), real-time RSSI graph, active channel, failover time, packets rescued vs lost, protected-vs-unprotected comparison, tappable packets, and mode + jam controls.
  • Transport: MQTT over WebSockets to the Pi broker; the app subscribes to the unified feed and publishes control commands.
  • Cloud dashboard: the same telemetry viewable from anywhere via HiveMQ Cloud.
  • Laptop dashboard: a USB-serial control panel for reliable, network-independent demos.

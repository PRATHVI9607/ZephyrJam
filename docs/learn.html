Pokedex Entry 01

Start Here

JamShield is a jamming-resilient sensor node. Normal WiFi gadgets die the instant someone attacks their WiFi. JamShield detects the attack in a fraction of a second and hops to a different radio, so the data keeps flowing.

Think of it like a Pokemon that switches type when hit by a super-effective move. Attack its WiFi? It becomes a Bluetooth type. Block that too? It goes ESP-NOW. The data never stops.

The Node

An ESP32 chip that reads a sensor and sends the data. The star of the show.

The Jammer

A second ESP32 that attacks the WiFi (the villain), so we can prove the defense works.

The Pi

A Raspberry Pi: the base camp that collects the data and shows the app.

Who is this for? Someone who knows nothing about this project. By the end you will understand every wire, every file, and be ready for teacher questions. Read top to bottom, or jump around with the Pokedex on the left.
Pokedex Entry 02

The Big Picture

Three devices talk to each other. The Node sends sensor data. The Pi collects it and hosts the app + a cloud link. The Jammer attacks. Here is the whole system on one screen:

ESP32 NODE RASPBERRY PI (the hub) YOU +---------+ WiFi / MQTT +-------------------------------+ +-----------+ | sensor | =================>| Mosquitto (MQTT broker) |==>| Phone app | | + radios| ---- BLE -------->| receiver.py (3-channel) | | (PWA) | | | .... ESP-NOW ....>| SQLite log | +-----------+ +---------+ | PWA host + HiveMQ cloud |==>| Cloud | ^ +-------------------------------+ +-----------+ | deauth attack +---------+ | JAMMER | (2nd ESP32, attacks the node's WiFi) +---------+

The single most important idea: the node has three radios and a brain that decides which one to use. When WiFi gets jammed, the brain switches to the next radio automatically. That switch is called a failover.

One-line summary: attack the WiFi, the node hops to Bluetooth or ESP-NOW, and the sensor data survives. The app shows it happening live.
Pokedex Entry 03

The Hardware

ESP32 (x2)

A tiny, cheap (~$5) microcontroller with WiFi and Bluetooth built in. It is not a full computer; it runs one program you load onto it. We use two: one as the Node, one as the Jammer. Key facts:

  • Dual-core CPU at 240 MHz, ~520 KB of RAM (kilobytes, not gigabytes).
  • WiFi + Bluetooth LE radios on-chip. Also does ESP-NOW (an Espressif-only radio protocol).
  • GPIO pins to connect sensors, and an ADC (analog-to-digital converter) to read analog sensors like a light sensor.
  • Talks to your PC over USB (a USB-to-serial chip, usually CP210x, gives it a COM port).

Raspberry Pi 4

A real little Linux computer (credit-card size). It runs services 24/7: the MQTT broker, our Python receiver, the app web server, and the cloud bridge. It is the hub everything reports to.

LDR (Light Dependent Resistor)

A cheap light sensor. Its resistance changes with light. Wired as a voltage divider into an ESP32 ADC pin (GPIO34), so the node can read "how bright is it" as a number 0-4095. This is the "sensor data" the node sends. (Any sensor would do; light is just easy to demo by covering it.)

Why these parts? ESP32 = cheap radio-rich microcontroller, perfect for IoT. Pi = enough power to run servers. LDR = a real physical signal so the demo is not fake numbers.
Pokedex Entry 04

ESP32 101

An ESP32 runs one firmware (program) at a time. To change what it does, you compile new firmware and flash it (write it into the chip's flash memory). There are three popular ways to write that firmware:

WayLanguageGood forWe use it?
ArduinoC++ (simple)Beginners, quick sketches, huge library setYes, for the ESP-NOW receiver sketch
ESP-IDFCFull ESP32 power, Espressif's official SDKNo (but it is the fallback for hard ESP-only features)
Zephyr RTOSCReal-time, multi-threaded, portableYes, the Node + Jammer run Zephyr
MicroPythonPythonLearning, fast experimentsNo (too slow/limited for real-time failover)

The key pins

  • EN (enable/reset) and GPIO0 (BOOT): the two "magic" pins. Holding GPIO0 low during reset puts the chip in download mode so you can flash it.
  • GPIO34: our LDR sensor input (ADC).
  • TX/RX: serial pins, routed through the USB chip so your PC sees a COM port at 115200 baud.
Mental model: ESP32 = a phone's radio chip with a small brain. You give it a job (firmware), it does that one job forever until you flash a new one.
Pokedex Entry 05

Flashing an ESP32

"Flashing" = copying the compiled firmware into the ESP32 over USB. The tool is esptool (a Python program). Steps:

  1. Plug the ESP32 into USB. It appears as a COM port (Windows: COM6, Linux: /dev/ttyUSB0).
  2. Build the firmware. That produces 3 files: bootloader.bin, partition-table.bin, zephyr.bin (the app).
  3. Run esptool to write them to the right memory addresses:
python -m esptool --chip esp32 -p COM6 -b 460800 write_flash --flash_size detect \
  0x1000  bootloader.bin \
  0x8000  partition-table.bin \
  0x10000 zephyr.bin

Those hex numbers (0x1000 etc.) are flash memory addresses: bootloader first, then the partition table, then the app. esptool resets the chip, uploads, verifies each block ("Hash of data verified"), and reboots it.

If it will not connect

  • Hold the BOOT button while esptool says "Connecting...", release after. That forces download mode.
  • Only one program can own a COM port at a time. Close serial monitors / Arduino IDE first.

In this project you never type that by hand: run.ps1 auto-detects the port, builds, and flashes for you. But now you know what it is doing under the hood.

Monitor the output: after flashing, open the serial port at 115200 baud to read the node's log (scripts/capture_win.py COM6). You will see it join WiFi, connect MQTT, and print STAT {...} lines.
Pokedex Entry 06

ESP32 + Python

Two different "Python + ESP32" ideas. Do not mix them up (a classic teacher trap):

1. Python that runs ON your PC to talk TO the ESP32

This is what we use. The ESP32 runs C firmware; Python on the laptop/Pi talks to it over USB serial or over the network. Examples in this project:

  • esptool (Python) flashes the chip.
  • pyserial reads the node's STAT lines over USB (used by dashboard.py, detect_ports.py, capture_win.py).
  • paho-mqtt (Python) receives the node's MQTT data on the Pi (receiver.py).
# Read a sensor value from the ESP32 over USB in plain Python:
import serial, json
s = serial.Serial("COM6", 115200, timeout=1)
while True:
    line = s.readline().decode("utf-8", "ignore").strip()
    if line.startswith("STAT "):
        data = json.loads(line[5:])
        print("light:", data["ldr"], " channel:", data["ch"])

2. Python that runs ON the ESP32 (MicroPython)

MicroPython is a tiny Python that runs directly on the chip. Great for learning. You would flash the MicroPython firmware, then upload .py files. Example of what an ESP32 sketch looks like in MicroPython:

# This runs ON the ESP32 (MicroPython), not the PC:
from machine import ADC, Pin
import time
ldr = ADC(Pin(34)); ldr.atten(ADC.ATTN_11DB)
while True:
    print("light =", ldr.read())   # 0..4095
    time.sleep(1)
Why we did NOT use MicroPython for the node: the node needs real-time behavior (detect a jam in 100 ms, run three radios, switch instantly). Python on a microcontroller is too slow and not real-time enough. That is why the node is written in C on Zephyr. MicroPython is perfect for learning, not for this job.
Pokedex Entry 07

Zephyr RTOS

Zephyr is a tiny real-time operating system for microcontrollers. Not Linux. It fits in kilobytes and its whole job is doing the right thing at the right time, every time. The node firmware runs on Zephyr.

Threads + a priority scheduler

Instead of one big loop, the program is split into independent threads. A scheduler runs them by priority: lower number = higher priority, and a high-priority thread instantly interrupts (preempts) a lower one. JamShield's threads:

ThreadPriorityPeriodJob
jam_detect2 (highest)100 msWatch RSSI + loss, run the jam state machine
sensor51000 msRead the light sensor
payload6500 msBuild + send each packet
control / button7 (lowest)eventUSB commands / BOOT button

So if a packet is being sent (priority 6) and a jam needs detecting (priority 2), the scheduler pauses the send, runs detection, then resumes. That guarantee is why detection is always on time.

Other Zephyr concepts you should name-drop

  • Devicetree (the .overlay file): describes hardware as data. The code asks for the "LDR ADC channel" by role, not by raw pin, so the same code works on other boards.
  • Kconfig (prj.conf): turn features on/off with CONFIG_WIFI=y, CONFIG_BT=y. Only enabled features get compiled in, so the image stays small.
  • west: the build tool. west build compiles, west flash loads it.
  • Static memory + integer math: no malloc in threads, no floating point in high-priority threads (it can corrupt the stack on this chip). That is why light is stored as lux_x10 (lux times 10, an integer).
  • Safe sharing: threads share data using a mutex (the LDR reading) and atomics (the sequence counter), so nothing corrupts.
Why Zephyr over Arduino here? Arduino gives you one loop() and global state. Zephyr gives real threads with guaranteed timing, plus full WiFi + Bluetooth stacks. For a device that must detect an attack and switch radios in milliseconds, you need that determinism.
Pokedex Entry 08

The 3 Radios (Types)

The node can send over three different radios. The brain (bearer manager) uses them in priority order. Think of them as the node's three types:

WiFi

Primary. Long range, fast, needs an access point (router). Sends data as MQTT over the internet/LAN. This is what gets jammed.

BLE

Bluetooth Low Energy. Short range, no router needed, direct to the Pi. Immune to WiFi deauth attacks. First fallback.

ESP-NOW

Espressif's connectionless radio: no router, no pairing, just broadcast. Hardest to jam. Last resort.

The order of failover is WiFi → BLE → ESP-NOW. Each is harder to knock out than the last. Every packet carries a sequence number so we can prove none were lost during a switch.

Protocol hopping (the whole trick)

"Protocol hopping" just means: if the current radio fails, jump to the next working one. The node keeps its WiFi connection alive even while sending on BLE, so it can tell the moment WiFi comes back and hop home.

ESP-NOW note: the Pi cannot receive ESP-NOW (it has no ESP32 radio). So for the demo the node also mirrors its ESP-NOW data over MQTT so the app can show it. The real ESP-NOW frames still go out over the air.
Pokedex Entry 09

MQTT

MQTT is the messaging language of IoT. It is a publish/subscribe system with a middleman called a broker (we run Mosquitto on the Pi).

  • Publish: a device sends a message to a named topic (like jamshield/sensor/ldr).
  • Subscribe: any device that wants those messages subscribes to that topic. The broker delivers.
  • Senders and receivers never talk directly; the broker connects them. Add a new listener any time without changing the node.

The topics in this project

TopicWho publishesWhat
jamshield/sensor/ldrNode (WiFi)Sensor telemetry JSON
jamshield/sensor/espnowNode (ESP-NOW mirror)ESP-NOW telemetry
jamshield/feedPi receiverUnified feed (WiFi+BLE+ESP-NOW) for the app
jamshield/controlApp / dashboardCommands: JAM, CLEAR, HOP, NOHOP, NOBLE

QoS (Quality of Service)

MQTT can guarantee delivery. We use QoS 1: every message gets an acknowledgement (PUBACK) back from the broker. The node counts sent vs acked, and missing acks = packet loss = a signal that WiFi is being jammed. Clever reuse.

WebSockets

Browsers cannot speak raw MQTT. So Mosquitto also listens on port 9001 as WebSockets, and the phone app uses mqtt.js to connect over ws://pi-ip:9001. Same MQTT, browser-friendly transport.

Pokedex Entry 10

Jamming the attack

WiFi has a weakness: management frames are not authenticated on WPA2 without PMF. One type, the deauthentication (deauth) frame, means "you are disconnected, leave now." A device cannot tell a real one from a fake one, so it just obeys.

What the Jammer does (3 steps)

  1. Joins your WiFi like a normal device, to learn the router's ID (BSSID) and lock onto its channel.
  2. Forges deauth frames that look like they came from the router.
  3. Floods them thousands per second (esp_wifi_80211_tx). The node gets kicked off, tries to reconnect, gets kicked again, forever. That is the jam.

This is protocol jamming (tricking devices with fake control messages), not noise jamming (blasting raw radio energy). Tiny power, pure cleverness.

Legal + ethical: deauth attacks are indiscriminate and illegal on any network you do not own or lack written permission to test. Point it only at your own access point. Modern WiFi (WPA3, or WPA2 with 802.11w / PMF) signs management frames, so a forged deauth is ignored and the attack fails. Mentioning that in a viva shows you understand the defense.

The two jam buttons

  • RF jammer (2nd ESP32): the real over-the-air attack above.
  • App / USB "logical" jam: a software flag that tells the node "act jammed." No frames sent, but the node runs its real detection and failover. Reliable for demos; the attack is simulated but the response is 100% real.
Pokedex Entry 11

Detect & Failover

The detector thread samples every 100 ms and watches two signals at once to avoid false alarms:

  1. RSSI: WiFi signal strength (in dBm, like -40 = strong, -85 = weak). A jam makes it collapse.
  2. Packet loss %: from the missing MQTT acks (see QoS above).

Only when both look bad (or a logical jam is forced), and it stays bad for a few samples in a row, does it raise the alarm. It runs a state machine:

CLEAR --degraded--> SUSPECTED --confirmed (3+ samples)--> CONFIRMED ^ | +---------------- RECOVERING <-- signal + acks return -------+

When it hits CONFIRMED, it calls the bearer manager which does the failover: WiFi → BLE (or → ESP-NOW in NO-BLE mode). While failed over, it keeps watching WiFi; when it recovers, it hops back home.

The three modes (for the demo)

ModeOn jamResult
HOPWiFi → BLE → ESP-NOWData saved (the win)
NO-HOPStay on WiFiData lost (shows the problem)
NO-BLEWiFi → ESP-NOW (skip BLE)Data saved via ESP-NOW
The whole demo in one sentence: HOP + jam keeps the data alive on another radio; NO-HOP + jam loses it. Same attack, with and without JamShield.
Pokedex Entry 12

The Raspberry Pi Hub

The Pi runs several always-on services (each auto-starts on boot):

ServiceJob
MosquittoMQTT broker. TCP port 1883 (devices) + WebSocket port 9001 (the app).
jamshield-recv (receiver.py)Receives all 3 channels (WiFi via MQTT, BLE via bleak, ESP-NOW mirror), logs to SQLite, and republishes everything to jamshield/feed.
jamshield-pwaA tiny web server that hosts the phone app on port 8081.
cloud bridgeMosquitto forwards the data up to HiveMQ cloud over TLS.

Why unify everything into one jamshield/feed topic? So the app has one simple thing to subscribe to and can watch the channel flip WiFi → BLE → ESP-NOW without caring how each arrived.

The Pi's Bluetooth was disabled by default on the fresh image; we re-enabled it (removed dtoverlay=disable-bt, installed pi-bluetooth, rebooted) so BLE reception works. Good "gotcha" story for a viva.

Pokedex Entry 13

The App (PWA)

A PWA (Progressive Web App) is a website that installs like an app. Ours lives in dashboard/pwa/ and connects to the Pi's MQTT over WebSockets. It shows the story, not just numbers:

  • Status: LINK SECURE / ATTACK DEFEATED / DATA LOST.
  • Live RSSI graph: a real radio measurement that moves as you move the node.
  • Active channel: WiFi / BLE / ESP-NOW, flips live during a jam.
  • Metrics: failover time (outage gap), packets rescued, packets lost.
  • Impact: protected (HOP) vs unprotected (NO-HOP) side by side.
  • Attack timeline + tappable packets (tap a packet to see its full JSON + which protocol carried it).
  • Controls: mode buttons + a jam button (publishes to jamshield/control).

It is three files: index.html (layout + CSS), app.js (MQTT logic), sw.js (service worker so it installs + updates). No build step, no framework, just vanilla JavaScript talking MQTT over WebSockets.

Open it: on a device on the same WiFi as the Pi, go to http://<pi-ip>:8081 and Add to Home Screen.
Pokedex Entry 14

The Cloud

To satisfy the "IoT cloud" requirement, the Pi forwards data to HiveMQ Cloud (a free hosted MQTT broker). This is a Mosquitto bridge: the Pi dials out over TLS (port 8883) and mirrors the topics up. No inbound ports opened on the Pi (safe).

  • Edge-first, cloud-optional: everything works locally even with no internet; the cloud is a bonus view from anywhere.
  • One gotcha we hit: a Mosquitto-to-non-Mosquitto bridge needs try_private false, or the cloud accepts then instantly drops the connection.

So all three required IoT boxes are ticked: MQTT (the whole spine), a Web/Mobile app (the PWA), and an IoT cloud (HiveMQ).

Pokedex Entry 16

Run It Yourself

  1. Power the Pi. It auto-joins WiFi and starts all services (~1 min).
  2. Power the Node (and Jammer, if using RF).
  3. On the laptop (both ESP32s on USB), run one command: 
    cd C:\Workspace\IotELL
.\run.ps1            # detect, build, flash both, open dashboard
.\run.ps1 -rf        # RF-jam scenario (WiFi to BLE)
.\run.ps1 -now       # logical-jam scenario (WiFi to ESP-NOW)
  4. On your phone (same WiFi), open http://<pi-ip>:8081.
  5. Demo: HOP mode + jam → channel flips to BLE, data saved. NO-HOP + jam → data lost.
Network changed? When WiFi/DHCP changes, the Pi's IP changes and the node's baked-in broker IP goes stale. run.ps1 auto-detects the Pi IP and reflashes. If it cannot find it: .\run.ps1 -PiIP 10.x.x.x.
Is it alive? ping prathvi.local for the Pi; the app header says NO BROKER LINK (wrong WiFi), WAITING FOR NODE (node off), or LINK SECURE (working).

Teacher Questions + Small Changes

Examiners love "change this live." Here is exactly where each change lives, so you can do it on the spot and sound like you own the code.

If they ask...Change this
Send data faster / slowerPAYLOAD_PERIOD_MS in payload_thread.c (500 ms default).
Use a different WiFiJS_WIFI_SSID / JS_WIFI_PSK in jamshield.h.
Change the MQTT topicJS_MQTT_TOPIC in jamshield.h.
Add a field to the packetThe snprintf in payload_thread.c (and the app if you want to show it).
Make detection more/less sensitiveMIN_CONFIRM_SAMPLES in jam_detect.c, or thresholds in adaptive_threshold.c.
Change the jam auto-expiry (15 s)The force-jam timeout in wifi_mqtt.c.
Change the BLE device nameJS_BT_DEVICE_NAME in jamshield.h.

Change the sensor pinThe .overlay file (devicetree), then rebuild.
Point the jammer at another AP.\run.ps1 -JamSsid "MyAP" -JamPsk "..." (a network you own).

Concept questions they may ask (short answers)

  • Why not just use WiFi? One radio = one point of failure. Jam it and the sensor goes dark. Multiple radios = resilience.
  • Is the jamming real? The RF jammer is real deauth. The app button is a logical trigger (simulated attack, real failover). Say which one you are using.
  • Why does it only work on your WiFi? WPA2 without PMF does not sign management frames. WPA3 / 802.11w defeats deauth.
  • Why Zephyr not Arduino? Real-time threads + guaranteed timing for detection, plus full WiFi and BLE stacks.
  • How do you know no data was lost? Every packet has a sequence number; gaps = loss. During HOP failover the sequence stays continuous.
  • How does the node detect a jam? Two signals: RSSI drop AND MQTT ack loss, sustained for a few samples. Dual-metric avoids false alarms.
  • Where is the cloud? HiveMQ Cloud via a Mosquitto TLS bridge from the Pi.
